Javascript frameworks

Gebruik

Het gebruik van frameworks voor javascript heeft voor- en nadelen. Uit eigen ervaring weet ik dat ik voor kleine projecten vaak gebruik maak van frameworks zoals jQuery. jQuery is super handig te gebruiken als je snel en makkelijk resultaat wil bereiken. Handig voor bijvoorbeeld animaties en event handlng, waar je anders een stuk meer tijd aan kwijt bent. Het gevaar is echter dat hoe meer frameworks/libraries je gaat implementeren in je applicatie, de performance vaak achteruit gaat. Het is dus niet altijd beter om voor frameworks te kiezen.

Voordelen

  • Efficiëntie

    Waar je normaal uren mee bezig bent kan met frameworks binnen enkele minuten worden geregeld door pre-built functies. Het developen wordt een stuk makkelijker en sneller gemaakt en wat dat betreft dus efficiënter.

  • Beveiliging

    In frameworks wordt tot op zekere hoogte al rekening gehouden met beveiliging. Een groot voordeel is ook de achterban van de frameworks, waar een groot aandeel testers bij zitten. Als je ergens niet uitkomt, of je ontdekt een kwetsbaar element in je applicatie dan kun je terecht op de bijbehorende website of het forum daarvan.

  • Kosten

    De meeste frameworks zijn gratis te gebruiken en helpen de ontwikkelaar om sneller resultaat op te leveren. Als de klant akkoord gaat met het voorstel om het framework te gebruiken kan dit minder tijd kosten en dus ook minder geld.

  • Support

    Frameworks hebben meestal goede referenties/communities en documentatie waar je terecht kunt om antwoorden te zoeken.

Nadelen

  • Je leert het framework, geen Javascript

    Het grootste nadeel is dat frameworks anders geschreven zijn dan de taal waar het op is gebaseerd. Je leert dus het framework en niet de programmeertaal die erachter zit. Als je het framework kent, dan kan je dus nog geen Javascript. Als je ooit een project moet doen die in Javascript geschreven is dan kom je in de problemen.

  • Limitaties

    De core van een framework kan niet worden aangepast. Dat betekent dus dat je beperkt wordt in de mogelijkheden die er worden aangeboden. Je moet dus een goede keuze maken voor het framework dat het beste bij je project aansluit.

  • Code is voor iedereen toegankelijk

    Het framework is voor iedereen bereikbaar en dat zorgt er ook voor dat mensen met slechte intenties erbij kunnen. Je kunt het bestuderen en waar nodig manipuleren op een kwaadaardige manier.

Single page web app

Wat is een single page application (SPA)?

Een Single Page Application (SPA), ook wel een Single Page Interface (SPI) genoemd, is eenwebapplicatie of website die past op een enkele webpagina met het doel een vlotte gebruikerservaring te bieden, die te vergelijken is met een computerprogramma dat op de computer van de gebruiker zelf draait. Er zijn diverse technieken beschikbaar om een Single Page Application te bouwen, waarin HTML-scripting een essentiële rol speelt. In een SPA wordt alle benodigde code - HTML, JavaScript en CSS - opgehaald met een enkele laadactie van de pagina, of de noodzakelijke middelen worden dynamisch geladen en aan de pagina toegevoegd zodra ze nodig zijn, meestal als reactie op acties van de gebruiker.

Voordelen

  • Single page application werkt voor responsive sites

    Het werkt goed voor responsive sites, omdat elk device maar 1 pagina hoeft in te laden.

  • Geen extra queries naar de server om pagina’s te laden

  • Content simpel, makkelijk en bruikbaar voor de gebruiker

    Single page webpagina’s zorgen voor een simpel lineair verhaal. Er is een begin, midden en een eind. Het scrollen maakt het ook toegankelijk voor mobile users. Scrollen door informatie is makkelijker dan het klikken door eindeloze links.

  • Hogere conversieratio

    Experts zeggen dat er meer conversie wordt behaald door middel van SPA’s in vergelijking tot grotere multi-page websites. Een studie van 37signals wijst uit dat een single landing page leidt tot 37,5 procent meer aanmeldingen dan een multi-page versie.

Nadelen

  • Snelheid

    Een uitdaging bij het maken van een SPA is de snelheid van het laden van de webpagina, omdat het framework en het renderen van de views in één keer moet gebeuren. Performance-wise kan dit verbeterd worden door middel van slimme caching en lazy-loading modules, maar het verschil blijft nog steeds groot vergeleken met multi-page websites.

  • Dynamic navigation handling

    In een SPA worden modules dynamisch geladen. Hierdoor is het mogelijk om de states van de applicatie te veranderen zonder dat de browser dit herkent. Dit zorgt ervoor dat je browser history niet meekrijgt dat je van state bent veranderd en raakt je navigatie in de knoop. Dit wordt netjes opgelost in moderne browsers door nieuwe methodes als history.pushState(), maar oudere browsers supporten deze methodes nog niet. Dit valt dan op te lossen door hash-based routing, maar hier moet dus wel rekening mee worden gehouden.

  • SEO

    In het verleden waren er veel zorgen over de SEO bij het kiezen voor een SPA. Maar met de toename van de keuze voor een SPA is dit steeds minder een issue. Google is in staat om alle informatie van een SPA te crawlen zonder enige issues en er zijn steeds meer engines die hier rekening mee houden. Er zijn nog zoekmachines die dit niet doen, maar het is de keuze van developers of het cruciaal genoeg is voor de user experience om de voordelen van een SPA te benutten.
